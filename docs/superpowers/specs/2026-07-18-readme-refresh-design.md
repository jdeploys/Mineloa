# Nnote README 개편 설계

## 목표

README를 처음 보는 사용자가 Nnote의 용도, 다운로드 상태, 처리 방식, 개인정보 경계를 빠르게 이해하고, 개발자는 같은 문서의 후반부에서 정확한 개발·검증·패키징 명령을 찾을 수 있게 한다.

## 변경 범위

- `README.md`의 정보 위계와 내용을 현재 구현에 맞춰 전면 정리한다.
- 최신 Airbnb 리디자인 스크린샷을 대표 이미지로 사용한다.
- GitHub의 `v0.0.1` 프리릴리스와 Windows/macOS 산출물을 안내한다.
- 기본 OpenAI 처리, 로컬 Whisper 전사, Codex CLI 요약의 차이를 설명한다.
- 개인정보, 로컬 저장, `.nnote` 이동 형식을 설명한다.
- 개발, 테스트, 패키징 명령과 관련 상세 문서를 연결한다.

다음은 변경하지 않는다.

- 앱 코드와 동작
- 빌드 및 릴리스 설정
- 기존 스크린샷과 상세 문서
- GitHub Release 자산

## 독자와 정보 위계

README는 사용자 안내를 먼저, 개발자 안내를 뒤에 둔다.

1. 제품 소개와 대표 스크린샷
2. 핵심 기능
3. 다운로드와 프리릴리스/서명 경고
4. 처리 방식 비교
5. 빠른 사용 순서
6. 개인정보와 파일 형식
7. 개발과 자동 검증
8. 패키징과 상세 문서

## 콘텐츠 원칙

- 현재 코드와 Release에서 확인한 사실만 단정한다.
- `FFmpeg 미지원`처럼 현재 구현과 모순되는 문장을 제거한다.
- OpenAI를 기본 처리 방식으로, 로컬 Whisper와 Codex CLI를 고급 선택지로 설명한다.
- 로컬 Whisper는 번들 Whisper/FFmpeg 실행 파일과 다운로드 모델을 사용한다는 점을 명시한다.
- Codex CLI는 별도 설치와 인증이 필요하며 전사문이 Codex 계정으로 전송됨을 명시한다.
- `v0.0.1`은 프리릴리스이며 산출물이 코드 서명되지 않았다는 경고를 다운로드 앞에 둔다.
- macOS App Store 제출 절차와 상세 릴리스 검증은 README에 복제하지 않고 기존 문서로 연결한다.
- 기능 목록은 유지보수 가능한 수준으로 제한하고 내부 구현 세부사항은 필요한 경우에만 적는다.

## 다운로드 표현

대표 다운로드 링크는 GitHub Release 페이지 하나로 통합한다. Windows x64, macOS Apple Silicon, macOS Intel 자산이 있다는 점을 표시하되, 현재 빌드는 프리릴리스이고 서명되지 않아 SmartScreen 또는 Gatekeeper 경고가 발생할 수 있다고 같은 영역에서 알린다.

## 처리 방식 표

| 단계 | 기본 | 고급 선택지 | 데이터 경계 |
| --- | --- | --- | --- |
| 전사 | OpenAI | 로컬 Whisper | OpenAI는 오디오 전송, Whisper는 기기에서 처리 |
| 요약 | OpenAI | Codex CLI | 선택한 공급자에 전사문 전달 |

OpenAI API 키는 OS 자격 증명 저장소에 보관되고 내보내기 파일에 포함되지 않는다는 설명을 표 아래에 둔다.

## 검증

문서 변경이므로 다음을 확인한다.

- README 내부의 상대 링크가 모두 존재한다.
- Release URL이 실제 `v0.0.1` 프리릴리스를 가리킨다.
- README에 미완성 표기나 예시용 값이 남아 있지 않다.
- `package.json`의 명령 및 실제 공급자 ID와 README 설명이 일치한다.
- `git diff --check`가 통과한다.
- 변경 범위가 README와 해당 설계/계획 문서에 한정된다.
